# Source Code vs Bytecode vs Binary Code

> **Example:** `console.log("Hello World!");` saved in `01_HelloWorld.js`

---

## TL;DR

| Layer | What it is | Who reads it | Example |
|-------|-----------|-------------|---------|
| **Source Code** | Human-written text in a programming language | Developers | `console.log("Hello World!");` |
| **Bytecode** | Intermediate instructions for a Virtual Machine | VM interpreter (e.g., V8 Ignition) | A sequence of opcodes like `LdaConstant`, `Call` |
| **Binary Code** | Raw CPU instructions (1s and 0s) | The processor hardware | `b8 01 00 00 00 ...` (x86-64) |

**Key takeaway:** Source → Bytecode → Binary is a **progressive lowering** from what humans write to what machines execute. JavaScript skips straight to bytecode at runtime — no separate compile step needed.

## 1. Full Comparison Table

| Aspect | Source Code | Bytecode | Binary Code (Machine Code) |
|--------|-------------|----------|----------------------------|
| **Definition** | Human-readable code written in a programming language | Intermediate representation — platform-independent instructions for a VM | Low-level instructions directly executed by the CPU; platform-specific |
| **Readability** | ✅ High — plain text with clear syntax | ⚠️ Low — numeric opcodes, requires VM knowledge to read | ❌ Not human-readable — raw hex or binary |
| **Who writes it** | Developers (you) | The compiler / interpreter (V8, JVM, CLR) | The compiler backend / assembler (TurboFan, GCC, LLVM) |
| **Execution** | Not executed directly — must be parsed first | Interpreted by a VM, may be JIT-compiled to native code | Runs directly on the CPU at full speed |
| **Speed** | Slowest (needs parsing + interpretation) | Medium (VM interprets with optimizations) | Fastest (no intermediate layer) |
| **Portability** | ✅ Portable — needs only a compatible runtime | ✅ Portable — any platform with a compatible VM runs it | ❌ Tied to a specific CPU architecture (x86, ARM, RISC-V) |
| **Storage size** | Small (this file is ~30 bytes) | Medium — larger than source, smaller than binary | Larger — includes full opcodes, registers, addressing |
| **Can you edit it?** | ✅ Yes — open in any text editor | ⚠️ Possible with a hex editor or decompiler, but impractical | ❌ Very difficult without reverse-engineering tools |
| **Deployed as?** | Rarely deployed raw in compiled langs; common in interpreted langs | Often the distribution artifact (`.class`, `.pyc`, `.wasm`) | The final shipped executable (`.exe`, `.dll`, `.so`, `.app`) |
| **Tooling example** | VS Code, Vim, WebStorm | V8 Ignition, JVM, Erlang BEAM | TurboFan (V8), GCC, MSVC, LLVM |

## 2. How Our Example Breaks Down

### Source Code Layer

```javascript
console.log("Hello World!");
```

This is **what you write**. It's:
- A member access (`console.log`)
- A function call with a string literal argument (`"Hello World!"`)
- A statement terminator (`;`)

### Bytecode Layer (what V8's Ignition produces internally)

After parsing, V8 generates something like this (simplified Ignition bytecode):

```
LdaConstant        r0, <"Hello World!">      // Load the string constant
Star               r1                         // Store it in register r1
LdaGlobal          r2, "console"              // Load the global `console` object
Star               r3                         // Store in register r3
LdaNamedProperty   r4, r3, "log"              // Load `log` property off `console`
Call               r5, r4, r1                 // Call `.log` with the string argument
Return                                      // Return undefined
```

Each line is a **bytecode instruction** — a human-readable mnemonic (like assembly, but for the V8 VM). Internally these are stored as compact 1-byte opcodes.

> **Real V8 bytecode for this** (you can dump it with `node --print-bytecode 01_HelloWorld.js`):
>
> ```
> [generated bytecode for function: (top-level)]
> Parameter count 1
> Frame size 8
>          0x2d8d00026e7e @    0 : 12 00             LdaConstant [0]
>          0x2d8d00026e80 @    2 : 1e f9             Star r1
>          0x2d8d00026e82 @    4 : 13 01 02          LdaGlobal [1], [2]
>          0x2d8d00026e85 @    7 : 1e f8             Star r2
>          0x2d8d00026e87 @    9 : 29 f8 03 04       LdaNamedProperty r2, [3], [4]
>          0x2d8d00026e8b @   13 : 1e f7             Star r3
>          0x2d8d00026e8d @   15 : 62 f7 f9 05       Call r3, r1, [5]
>          0x2d8d00026e91 @   19 : 04                Return
> Constant pool (size = 4)
> [0]: "Hello World!"
> [1]: "console"
> [2]: "log"
> ```

### Binary Code Layer (what the CPU actually runs)

When V8's TurboFan JIT decides this code is "hot" (executed many times), it compiles to raw machine code. For x86-64, that might look like:

```asm
; Simplified x86-64 assembly for console.log("Hello World!")
mov       rdi, [r14 + console_offset]    ; load console object
lea       rsi, [rip + hello_world_string] ; load string address
call      [rdi + log_offset]             ; call the log function
```

Which encodes to machine bytes:

```
48 8b 3e          mov rdi, [rsi]
48 8d 35 XX XX XX XX   lea rsi, [rip + offset]
ff 57 20          call [rdi + 0x20]
```

These bytes are sent directly to the CPU's instruction decoder — no further translation needed.

## 3. Step-by-Step Walkthrough

| Step | What Happens | Format | Who Does It |
|------|-------------|--------|-------------|
| **①** | You write `console.log("Hello World!");` in a file | Source Code (UTF-8 text) | You (developer) |
| **②** | You run `node 01_HelloWorld.js` | — | Shell / OS |
| **③** | Node.js loads the file, V8 parses it into an **AST** (Abstract Syntax Tree) | Tree of nodes (`CallExpression`, `MemberExpression`, `StringLiteral`) | V8 Parser |
| **④** | V8's Ignition walks the AST and emits **bytecode** | Bytecode (opcodes + operands) | V8 Ignition |
| **⑤** | Ignition **interprets** the bytecode line by line — this already prints "Hello World!" to the console | — | V8 Ignition (Interpreter) |
| **⑥** | If this function runs many times, TurboFan **JIT-compiles** the bytecode to optimized **machine code** | Binary (x86/ARM instructions) | V8 TurboFan |
| **⑦** | Subsequent calls skip interpretation and run the native binary directly | — | CPU (hardware) |

> **Important:** For a one-shot script like this, the code **never reaches Step ⑥** — it's interpreted as bytecode only, which is fast enough. JIT compilation only triggers for "hot" (frequently executed) code paths.

## 5. Breakdown Table — What Each Layer "Sees"

| Layer | `console` | `.` | `log` | `(` | `"Hello World!"` | `)` | `;` |
|-------|-----------|-----|-------|-----|------------------|-----|-----|
| **Source Code** | Identifier `console` | Member access operator | Method name `log` | Open paren | String literal `Hello World!` | Close paren | Statement terminator |
| **Bytecode** | `LdaGlobal [1]` (load by constant pool index) | — (resolved at property access) | `LdaNamedProperty r2, [3]` (load by property name index) | — (implicit in `Call`) | `LdaConstant [0]` (load from constant pool) | — | — (implicit — bytecode is instruction-based) |
| **Binary Code** | Memory address of the global `console` object | Pointer offset to the `.log` function | Call instruction with offset | — | Pointer to the string in memory | — | — (CPU follows call/ret) |

## 6. Real-World Comparison

| Dimension | Source Code | Bytecode | Binary Code |
|-----------|-------------|----------|-------------|
| **Analogy** | A recipe written in English | A recipe translated to kitchen equipment commands ("oven to 350°", "whisk for 2 min") | The actual physical motion of your hands and the oven heating up |
| **Portability** | Same recipe works in any kitchen | Same commands work in any kitchen with the same appliances | The physical motion is specific to *your* kitchen layout |
| **Modification** | Cross out an ingredient | Re-program the oven timer | Change the wiring in the oven |
| **Tool** | Pen and paper | Translators / VMs | Brawns and machines (CPU) |

---

## Key Concepts

- **Source code** is the **what** — what you want the program to do
- **Bytecode** is the **how** (platform-agnostic) — a recipe the VM can follow
- **Binary code** is the **do it** (platform-specific) — raw instructions the CPU executes

JavaScript lives entirely at runtime: there's no separate compile step. V8 parses source → generates bytecode → interprets it immediately. Only if the same code runs many times does V8 invest the CPU time to JIT-compile it to optimized machine code.
